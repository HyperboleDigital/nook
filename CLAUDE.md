# Nook — Project Context for Claude

## What this app is
Nook is a SaaS for real estate agents. Three core features:
1. **3D Tours** — agent uploads a property walkthrough video → GPU processes it into a Gaussian Splat (.ply) → agent shares a 3D walkthrough link with clients
2. **Reels** — agent generates a cinematic 9:16 social media video from a property photo (via Higgsfield AI)
3. **Room Restyle** — agent uploads a room photo → taps a detected item on a full-screen, canvas-first editor → sources a replacement (a pasted retailer link, an inspo photo, or a description) or removes it from the room entirely → AI (Gemini "Nano Banana" image model) restages the room → each swapped item is matched to a real, buyable product ("shop the look") via Google Lens + retailer APIs, searched only AFTER generate (not the moment a photo is uploaded) → agent shares a client link with the same tappable, shop-the-look experience. Generate is fire-and-forget end to end (see below) — the whole thing runs server-side, so nothing is lost if the tab closes. This is the feature under active development.

Live app: https://nook-lime.vercel.app  
GitHub: https://github.com/HyperboleDigital/nook

---

## Stack
- **Framework**: Next.js 16 (App Router, `src/` dir, TypeScript)
- **Auth**: Clerk (`src/proxy.ts` is the middleware — Next.js 16 picks this up, NOT `middleware.ts`)
- **Database**: Supabase (DB only — no Supabase Storage, tables: `users`, `tours`, `reels`)
- **File storage**: Vercel Blob (videos up to 2 GB, PLY output files) — `BLOB_READ_WRITE_TOKEN` required
- **GPU processing**: Modal.com serverless GPU (`modal/worker.py`) — deployed as `nook-3dgs`
- **3D pipeline**: Nerfstudio `splatfacto` on Modal A10 GPU (~35 min, ~$0.73/scan)
- **3D viewer**: SuperSplat (`superspl.at/editor?load=<ply_url>`) — display only, not a processor
- **Reels**: Higgsfield AI API
- **Room Restyle**: Gemini image model (`gemini-2.5-flash-image` "Nano Banana", pro variant `gemini-3-pro-image-preview`) for compositing; `sharp` for image canonicalization; Google Lens + SerpApi / Unwrangle / Rainforest for product matching
- **Billing**: Stripe (plans: free / starter / pro)

**Pending / known issues:**
- Clerk keys are still `pk_test_` (dev mode) — needs production keys before real users
- Supabase Storage bucket `nook-uploads` still exists but is no longer used
- No usage/quota enforcement on free plan yet
- **`supabase/migrations/013_restyle_searches.sql` needs to be applied manually** (no Supabase CLI is configured in this project — migrations are pasted into the Supabase SQL editor by hand). Until it's run, `GET /api/restyle/[id]/searches` 500s harmlessly (the client swallows the error and just starts with an empty search cache) — search/pick still works, results just won't persist across reloads.
- **`supabase/migrations/014_restyle_room_type.sql` also needs manual application.** Adds a nullable `room_type` column to `restyles`, set by the capture wizard's room-type picker (`/restyle/new`). The insert in `POST /api/restyle` only includes `room_type` when a value was actually picked, so uploads work fine even before this migration is applied — the field is just silently dropped until then.
- **`supabase/migrations/015_restyle_placement.sql` also needs manual application — BEFORE deploying the pin-placement code.** Adds `restyle_edits.placement` jsonb (`{x, y, note}`, 0–1000 box_2d space, set by the add-item tap-to-place pin) and drops the dead `restyle_versions` table. Unlike 014, this one is not optional-at-runtime: `PATCH /api/restyle/[id]/edits` writes the `placement` column directly and will 500 on an unknown column.
- **`supabase/migrations/016_restyle_generating_status.sql` also needs manual application — BEFORE deploying the resumable-generate code.** Adds `restyles.generating_started_at`/`generate_error`, set/cleared by `POST /api/restyle/[id]/generate` so a fresh page load can detect an in-flight render and resume showing progress instead of it finishing invisibly. Not optional-at-runtime: the generate route writes these columns directly and will 500 on an unknown column.
- Restyle hotspots on a *render* are an approximation, not real detection: object positions are only ever detected on the original photo (`restyles.detected_objects`), so a hotspot on the styled result reuses that item's ORIGINAL box_2d position. Correct for the common case (a swap usually stays roughly where the original piece was) but can drift if Nano Banana repositions/resizes furniture significantly. A pinned "add" item uses a small box synthesized around its tap-to-place pin instead; an add that's still unpinned by the time it renders gets one auto-assigned (see the auto-locate gotcha below) — only a genuinely never-rendered add has no hotspot at all, and shows only in `ChangesPanel`.
- **An "add" edit the user never manually pinned still gets a hotspot once it renders — auto-located, not left to "Choose a spot" forever.** Nano Banana decides where to put an unpinned add on its own; `generate/route.ts`'s `after()` locates it after `recompose` finishes, via `locateItemInRoom` (`gemini.ts` — a targeted "find the {label} in this room photo" box query, same precedent as `locateProductPhoto`, best-effort/null-safe) run against the just-rendered image for every active `add` edit with `target_label` and a still-null `placement`. A hit PATCHes `restyle_edits.placement` to the detected box's center — the SAME field a manual pin writes — so `useRestyleWorkspace`'s existing add-hotspot code (which only checks `e.placement`) picks it up with no client-side changes; the client sees it on the next `GET /api/restyles/[id]` (`pollGenerating` already re-fetches `edits`). A miss just leaves it as before (still reachable via `ChangesPanel`'s "Choose a spot"). Only runs when `placement` is null, so it never overrides a user's manual pin, and is a no-op on a cache-hit re-render where placement was already resolved on a prior generate.
- **A free-text "add" label is normalized BEFORE it's accepted, never used raw.** `SourcePanel`'s "What are you adding?" step used to set `sourcing.label` directly from whatever the user typed — a full sentence like "floor plant next on the left of the tv" then became the label EVERYWHERE (menu title, canvas hotspot text, changes-rail card) forever, since `target_label` is set once at staging time and never re-derived. Now `confirmItemDraft` POSTs to `/api/restyle/normalize-label` first, which calls `normalizeItemLabel` (`gemini.ts`) to collapse it to a short canonical name ("Floor Plant") — and doubles as content moderation, since a non-item/gibberish/offensive input comes back as a 422 surfaced inline instead of silently becoming the label. The input also has a native `<datalist>` of common furniture/decor terms (`COMMON_ITEMS`, kept in sync with `DETECT_PROMPT`'s vocabulary) for guided autocomplete — free text is still accepted either way, just always passes through the same normalize call. `useRestyleWorkspace`'s `setSourcingLabel` itself does NOT normalize (it just sets whatever it's given) — normalization is `SourcePanel`'s job, at the one call site that accepts raw user text.
- **The four "Edit item" menu rows (`SourcePanel`'s `MenuRow`) are color-coded by tone, each with a slow-shifting gradient sheen** (`MENU_ROW_TONES` + the `icon-gradient-shift` keyframe in `globals.css`) — swap=violet/fuchsia, shop=emerald/teal, adjust=amber/orange, remove=red/rose (remove's red predates this and was the reason to extend the same treatment to the other three, so all four read as distinct categories at a glance instead of the destructive one being the sole exception). The keyframe needs no separate reduced-motion guard — `globals.css`'s existing wildcard `prefers-reduced-motion` block already kills all animations/transitions app-wide.
- **The shopping-bag icon means ONE specific thing everywhere it appears: this edit has a real `buy_url`.** It's never used for "this is placed" or "this is an add" generically — an add/swap sourced from a plain description or an inspo photo with nothing resolved yet is NOT shoppable, so it gets a different icon (a "new" sparkle on the canvas hotspot marker in `ObjectHotspots.tsx`; `Plus`/`Replace` as the `ChangesPanel` card fallback icon) instead of implying it's purchasable. The "Find similar items"/"Find buyable matches" actions (`SourcePanel`'s menu row, `ChangesPanel`'s inspo-card button) were renamed to "Shop similar items" with the same shopping-bag icon, so the icon's meaning ("there's a real product angle here") stays consistent between the STATE marker and the ACTION that gets you one.
- **Gemini can silently drop the `box_2d` key from a JSON box response** (`{"2d":[...]}` instead of `{"box_2d":[...]}`, observed with `responseMimeType: "application/json"` under `gemini-2.5-flash`) — a naive `parsed.box_2d` check reads this as "no box found" and discards a perfectly good detection with zero visible error (it's a silent `null`, not a thrown exception). This is exactly what made a new "add" item's auto-located hotspot never show up despite Gemini correctly finding it. Fixed once, centrally: `extractBox2d()` (`gemini.ts`) falls back to ANY 4-number array value in the parsed object if `box_2d` itself isn't present, and both `locateProductPhoto` and `locateItemInRoom` go through it now. Any FUTURE "ask Gemini for a JSON box" call should reuse `extractBox2d`, not hand-roll its own `parsed.box_2d` check.
- **ONE image, and every item taps into the SAME "Edit item" menu.** There is no version navigation and no "viewing the original" state: `ws.displayUrl` is ALWAYS `restyles.current_url` (the current generation). `previewUrl` and the version-history strip (`VersionsStrip`) are gone; the original photo + `restyle_renders` are pure backend machinery (recompose composites from the original; the per-signature render cache makes toggling a change back instant). Tapping ANY detected item — unchanged or already swapped/adjusted — routes through `RestyleCanvas`'s `handleTap` → `ws.openSourcing(label, mode, stagedEditId)` → the category menu (Swap it / Find similar / Adjust it / Remove it) in the rail (desktop) or sheet (mobile). There are **no on-canvas popovers** anymore — `HotspotPopover` and `QueuedHotspotPopover` were deleted; on/off toggles and Buy links live in the `ChangesPanel` rail, everything else in the menu. `canvasHotspots` still tags each item `idle`/`confirming`/`queued`/`placed` (used only for the at-rest marker style in `ObjectHotspots` now, not to pick a popover). A floating "+ Add" button drops a pin over the CURRENT image (not the bare original — that was a bug) for new items.
- **Popovers must clamp their horizontal position in PIXELS, not just percent of the image.** A percent-only clamp (e.g. `min(max(cx, 18), 82)`) assumes the rendered image is wide enough that half the popover's fixed width is a small percentage of it — that breaks on a narrower image (portrait, small viewport, side-by-side desktop) and the fixed-width card clips off the `overflow-hidden` frame. Fix pattern (see `PinPlacementLayer.tsx`, the only on-canvas floating card left): `left: clamp(HALF_WIDTH_PX, ${cx}%, calc(100% - HALF_WIDTH_PX))` — CSS `clamp()` compares mixed px/% units correctly.
- **Don't put drag-driven state in `useRestyleWorkspace` — it re-renders the entire editor tree.** The compare slider used to live in the shared hook and called `setCompare` on every raw `pointermove`, which re-rendered `ChangesPanel`/`GenerateBar`/etc. on every pixel of drag (the hook backs the whole `RestyleStudio` subtree, not just the canvas), which is what made it feel laggy. Fixed by moving `compare`/`dragging`/`imgWrapRef`/`sliderHandlers` into local state in `RestyleCanvas.tsx` (nothing else read them) and throttling `setCompare` to at most once per animation frame via a `requestAnimationFrame` guard. If something is only read by one component, keep its state there — don't default to putting it in the shared hook just because everything else lives there.
- **Toggling a change in `ChangesPanel` is a flag flip (`ws.toggle`) — it never itself calls Gemini.** An earlier version (`toggleAndRegenerate`) fired a full render on every switch flip and bounced the card between two separate rail sections ("Shop this look" ↔ a "Turned off" section at the bottom) — off-then-on cost two render cycles and two card jumps. Now a card's identity and position never change: its status chip (`in-room` / `pending` / `turning-off` / `off`, derived in `railEdits`) updates in place. Whether the room ACTUALLY updates right away depends on `adoptCachedRenderIfKnown` (see below) — a combination seen before adopts instantly and for free, a new one waits for Generate. The old "can't turn off the only active change" guard was removed entirely — batching means switching everything off just queues a change that Generate resolves to the bare original (the same outcome as "Start from original"), which is no longer a surprising collapse.
- **A toggle/delete/stage that lands back on an ALREADY-rendered combination adopts that image instantly — no Generate click needed.** `edits/route.ts` (POST/PATCH/DELETE) and `product/route.ts` (POST) all end by calling `adoptCachedRenderIfKnown(id)` (`restyle-render.ts`): it recomputes the resulting active-edit signature and, if a `restyle_renders` row already exists for it (or the set is empty, i.e. the bare original), updates `restyles.current_url` to that cached image right there in the mutation response — the client (`useRestyleWorkspace`'s `updateEdits(edits, current_url?)`) applies it immediately. A genuinely new combination just leaves `current_url` untouched, and the existing `pendingCount`/status-chip machinery already communicates "generate to see this." This is what makes toggling something off and back on (a `restyle_renders` cache hit either way, once it's been rendered once) free, while a first-time combination still costs a real Generate — exactly the distinction the honest-labeling status chips were built to communicate. Don't add a second "read the render list and set current_url" path outside this function — it's the one place server-side state is allowed to silently swap `current_url` without going through `generate()`/`pollGenerating`.
- **`restyle_renders`' cache cap (`MAX_RENDERS` in `restyle-render.ts`) was raised from 8 to 40** — 8 was easy to blow through in one active editing session (several swaps/removes/adds, each toggled a few times), silently evicting the exact combination a toggle-back needed and forcing a surprise real regenerate for something already rendered once. Blob storage for extra cached PNGs is far cheaper than an unnecessary Gemini call; still bounded, just much more generous.
- **`GenerateBar`'s badge/enabled-state is `ws.pendingCount`, not `activeEdits.length`.** Edits stay `active` after a successful render (nothing deactivates them), so counting active edits meant the badge read e.g. "3" forever even with nothing left to apply. `pendingCount` is the symmetric diff between the active-edit id set and the displayed render's `signature` (comma-joined active-edit ids, see `restyle-render.ts`) — it's 0 (badge hidden, button reads "Up to date") exactly when the room already reflects every active edit.

### Room Restyle — API routes (`src/app/api/restyle/`)
- `upload-url/route.ts` — POST, signed Vercel Blob upload token for room/inspo photos (`handleUpload`, mirrors `tours/upload-url/route.ts`). The browser uploads the actual image bytes straight to Blob using `@vercel/blob/client`'s `upload()`, decoupled from our own functions — closing the tab mid-transfer just loses that unfinished upload, never a half-created DB row. `route.ts` and `[id]/product/route.ts` below only ever receive the resulting blob URL as JSON.
- `route.ts` — POST, create a restyle project from an already-uploaded room photo (`{photoUrl, title?, room_type?}` JSON — no file bytes in this request). Fetches the blob, `sharp` canonicalizes (EXIF-rotate + downscale to 1536px), re-uploads the canonical copy, best-effort deletes the client's raw upload, inserts `restyles` row, then fires object detection in the background via Next's `after()` using the buffer already in memory (no refetch) — chips are usually ready by the time the editor loads instead of a separate client-triggered detect call.
- `detect/route.ts` — legacy/fallback path: Gemini detects editable objects in the room (JSON `{imageUrl,restyleId}` or multipart). The client only calls this if the background detection from `route.ts` hasn't landed after ~20s of polling. Caches `detected_objects` on the project (detection is non-deterministic, so it's persisted for a stable item list).
- `normalize-label/route.ts` — POST `{text}` → `{label}`, stateless (not restyle-scoped). Turns the free text typed into `SourcePanel`'s "What are you adding?" step into a short, clean item name via `normalizeItemLabel` (`gemini.ts`) BEFORE it's accepted as `sourcing.label` — see the label-normalization gotcha below. A 422 (non-item/gibberish/offensive input) is surfaced as an inline error rather than silently accepted.
- `[id]/product/route.ts` — POST, stage a product/photo as a reference edit. Input shapes, all JSON: `{url}` (pasted retailer link — a confirmed real product), `{token}` (SerpApi immersive token → resolved URL), or `{imageUrl}` (an inspo photo the client already uploaded to Blob via `upload-url/route.ts` — staged as a reference with **no buy link yet**; search is deferred, see below). `targetLabel` force-targets a specific slot; `replaceEditId` deletes a superseded edit (e.g. an inspo photo now replaced by a real pick) in the same request. Auto-decides replace-vs-add via `matchDetected`. Does NOT render. Shared staging logic (insert + single-active-per-label dedupe) lives in `src/lib/restyle-edits.ts`'s `stageEdit()`.
- `[id]/visual-search/route.ts` — POST, find buyable products via Google Lens + keyword search, Gemini-scored against the target image. Three input shapes: `image` (file), `imageUrl` (an already-staged/cropped photo — used for the deferred post-generate search, see below), or `query` (text, from "Describe it"). Responds fast with unscored results, then finishes Gemini scoring + resolves Wayfair immersive tokens in the background via `after()`, persisting the final set to `restyle_searches` (see DB tables). Does NOT render.
- `[id]/searches/route.ts` — GET, the persisted search results for a project (optional `?label=`). Replaces what used to be a client-side localStorage cache.
- `[id]/edits/route.ts` — POST add / PATCH toggle / DELETE a change layer (multipart). Does NOT render. POST's single-active-per-`target_label` dedupe covers `item`/`add`/`remove` together (a swap, an add, and a targeted remove of the same slot are mutually exclusive — staging any one deactivates the others). PATCH also accepts a batch `{ states: {editId: bool} }` map for atomically flipping several edits in one call (used by "Start from original").
- `[id]/generate/route.ts` — POST, **fire-and-forget**: applies an optional server-side edit-state change (`{toggle: {editId,active}}` or `{emptyRoom: true}` — atomic, replaces what used to be a client for-loop that died if the tab closed mid-sequence), marks `generating_started_at`, and responds `202` immediately. The actual render (`recompose`) **and** the deferred inspo product search (via `src/lib/restyle-search.ts`) both run inside `after()` — so the whole "make the image, then find buyable matches for anything that was just an inspo photo" sequence survives the client disconnecting entirely. `maxDuration = 300` (covers render + N searches). The client (`useRestyleWorkspace`'s `pollGenerating`) polls `GET /api/restyles/[id]` until `generating_started_at` clears.
- `[id]/items/route.ts` — item list helper (custom/not-detected items).
- `restyles/route.ts`, `restyles/[id]/route.ts` — list/read restyle projects (plural — the gallery/index endpoints). `GET /api/restyles/[id]` returns `{...restyle, edits, renders}` and is the single read both the initial load and generate-polling use.

### Room Restyle — pages/components
The editor (`[id]/`) lives in its own **`src/app/(studio)/restyle/[id]/`** route group — a chrome-free, full-viewport shell (see `(studio)/layout.tsx`) with no app sidebar/padding/max-width, so the room photo can go edge-to-edge. The gallery and capture wizard stay in the normal app shell: `src/app/(app)/restyle/page.tsx` (projects gallery) and `src/app/(app)/restyle/new/page.tsx` (upload/paste a room photo — clipboard paste supported; "Take a photo" only on phones, "Choose a photo" on desktop; client-side `downscaleImage()` before upload, see `src/lib/image-client.ts`; a camera-captured portrait photo triggers a "Landscape works best" retake prompt with a "Use it anyway" escape hatch — a web file-input camera can't force landscape orientation or the 0.5x lens, so this is a post-hoc check, not a real constraint). The URL is unchanged (`/restyle/[id]`) — route groups don't affect routing, and Clerk protection in `src/proxy.ts` is path-based so auth is unaffected by the move.
- `(studio)/restyle/[id]/page.tsx` — thin shell: `h-dvh flex flex-col` with a slim `h-12` top bar (back arrow, title input) + `<RestyleStudio>` filling the rest of the viewport.
- `(studio)/restyle/[id]/useRestyleWorkspace.ts` — the single state hook everything else reads from. Key concepts: `sourcing` (the item currently being edited — `{label, mode: "swap"|"add", view: "menu"|"compose"|"adjust"|"similar", stagedEditId, hasMenu, lastStaged?}`, cleared entirely on panel close so stale banners can't outlive the item they describe). Tapping ANY existing item → `openSourcing(label, mode, stagedEditId)` → `view:"menu"`, `hasMenu:true`; a fresh "+ Add" (pin-flow `placeAddLocation`/`skipAddLocation`) starts at `view:"compose"`, `hasMenu:false` (nothing to categorize). `hasMenu` drives the "← Back to menu" affordance in `SourcePanel`/`SimilarItemsPanel`. `setSourcingView(view)` navigates within an open session (menu ↔ compose/adjust/similar) without losing label/mode/stagedEditId — see the reframed-editing gotcha below. `searches` (per-label search state hydrated from `GET /searches` and updated in place as scoring finishes), `canvasHotspots` (unified hotspot list for whichever image is on screen — see gotchas). Picking a candidate (`pickCandidate`) is optimistic: the edit appears staged immediately using the search result's own thumbnail/title/price, reconciled or rolled back once the server responds. `toggle` flips one edit's `active` flag alone and PATCHes it — no regenerate, that only happens on Generate (see the batch-toggle gotcha above; this replaced a former `toggleAndRegenerate`). `deactivateAll()` batch-PATCHes every active edit off in one call (used by "Start from original" — replaces a former client for-loop). `stageRemove(label)` stages a targeted `kind:"remove"` edit for a detected item (see "per-object remove" gotcha below). `stageRefine(label, instruction)` stages a `kind:"refine"` edit — a free-text adjustment to an item ALREADY in the room ("mount it on the wall") rather than sourcing a replacement (see the custom-instructions gotcha below). `shownProductIds` (a render's `signature`-derived id set, or `null`/empty-set pre-render — see the honest-labeling gotcha) feeds two derived values consumed by `ChangesPanel`: `pendingCount` (the symmetric diff between active-edit ids and `shownProductIds` — what `GenerateBar`'s badge/enabled-state uses) and `railEdits` (`{edit, status}[]`, one persistent card per relevant `item`/`add`/`refine`/`remove` edit — active ones tagged `in-room`/`pending`, and inactive-but-still-relevant ones (superseded-label-excluded, deduped to the latest per label) tagged `turning-off`/`off`). `productEdits` is `railEdits` filtered to in-room + buy_url, feeding the canvas's floating shop-summary pill. The before/after compare slider's state (`compare`/`imgWrapRef`/`sliderHandlers`) is NOT in this hook — it lives locally in `RestyleCanvas.tsx` (see gotchas) since nothing else needs it. `generate(body?)` is fire-and-forget: POST returns 202 immediately, then `pollGenerating` (also used to resume an in-flight render found on page load, via `restyles.generating_started_at` — migration 016) polls `GET /api/restyles/[id]` every ~4s until it clears, adopts the fresh `current_url`/`edits`/`renders`, surfaces `generate_error`, and re-fetches `/searches` once to hydrate whatever the server-side deferred search produced while nobody was watching. `generating` is derived from `generatingStartedAt !== null`; `expectedSeconds` (`clamp(30 + 8×activeEdits, 30, 90)`) and `generatingStartedAt` are handed to `ProgressOverlay`, which does its own ticking (see gotchas — this used to tick in the hook and re-render the whole tree every 300ms).
- `(studio)/restyle/[id]/RestyleStudio.tsx` — layout owner, branches fully by breakpoint (not just CSS tweaks) since desktop and mobile have genuinely different structures. Desktop: `RestyleCanvas` fills an immersive `relative h-full w-full` stage; a **docked, ALWAYS-VISIBLE right rail** (the collapse-to-icon toggle that used to live here was removed — the rail is core to the editor, not optional chrome to tuck away) defaults to `ChangesPanel` (the same component before AND after a render — no more swap between two different components on first generate) and swaps to `SourcePanel`/`SimilarItemsPanel` while `sourcing` is open; `GenerateBar` floats bottom-center as a pill (`variant="floating"`). Mobile: stacked-column layout (canvas → `ChangesPanel` → sticky `GenerateBar`), sourcing in a bottom `Sheet`. (No version-history strip in either — see the one-image gotcha.)
- `(studio)/restyle/[id]/RestyleCanvas.tsx` — the ONE image the user edits (`ws.displayUrl` = current generation; see the one-image gotcha). - `shopping-search.ts` — SerpApi Google Lens visual match + Google Shopping/Amazon/Walmart/HD keyword search (always run in parallel, not gated behind a match-count threshold); `resolveImmersiveToken` resolves ONE Wayfair immersive token → a real URL. **Token resolution is now LAZY (a SerpApi-cost decision):** `resolveTokenUrls` (which resolved every token-only candidate in the background) is no longer wired into the search flow — the product route resolves a token only when that specific candidate is actually picked (`product/route.ts`'s `{token}` branch). This cut SerpApi usage substantially (most candidates are never picked); the only tradeoff is a token-only Wayfair card shows no pre-resolved "View on X" link until picked. ## Room Restyle flow (end-to-end)
1. Agent goes to `/restyle/new`, uploads or pastes a room photo → browser uploads it directly to Vercel Blob (`POST /api/restyle/upload-url` for the signed token, then `@vercel/blob/client`'s `upload()`) → `POST /api/restyle` `{photoUrl}` (sharp canonicalizes, stores original, creates project, fires detection in the background) → redirect to `/restyle/[id]`. Splitting the byte transfer from project creation means closing the tab mid-upload just loses the unfinished Blob transfer, never a half-created project.
2. Immersive full-viewport editor loads (`(studio)/restyle/[id]`): room photo edge-to-edge with tappable hotspot circles (real detected positions). Right rail (floating on desktop, inline on mobile) shows "Queued changes" (empty at first).
3. Agent taps a hotspot. Sourcing panel opens (floating rail on desktop, bottom sheet on mobile):
   - **Paste a link** → `POST /api/restyle/[id]/product` `{url}` stages it immediately as a confirmed real product (buy link known right away, no search needed).
   - **Upload a photo** → same direct-to-Blob upload as step 1, then `POST /api/restyle/[id]/product` `{imageUrl}` stages it as **inspo only** — no shopping search runs yet. This is deliberate: search used to fire the instant a photo was picked, spending API/token cost before the user had even seen the room or decided the item was worth shopping for.
   - **Describe it** → manual "Find" triggers `POST /api/restyle/[id]/visual-search` (text `query`).
   - **Remove it** → `ws.stageRemove(label)` stages a targeted `kind:"remove"` edit for that detected item — a distinct action from un-toggling a swap (which just reverts to the original item); this actually erases it from the room.
4. Agent hits Generate → `POST /api/restyle/[id]/generate` responds `202` immediately (fire-and-forget); server-side in `after()`: `recompose` composites all active edits with Gemini and uploads the render (cached per active-edit signature), **then the deferred search happens**: for every active edit that's inspo-only (a reference photo, no buy_url), `runProductSearch`-style lookup (via `restyle-search.ts`) finds real buyable matches — scoped to whatever actually made it into the render. The client polls until this all finishes, showing a self-ticking progress overlay (% + rotating warm messages + estimated seconds) the whole time — closing the tab mid-generate loses none of this, it all completes server-side and rehydrates on the next load.
5. Canvas now shows the render. Hotspots reappear (positions approximated from each swapped item's ORIGINAL detected box — see gotchas; a placed *remove* shows no hotspot at all) with a popover (thumbnail/price/Show-similar/Buy) on tap. Right rail switches to "Shop this look": shoppable items as cards (retailer/price/buy link/running total), a "Removed" section with Restore for anything erased from the room, plus inline search results for anything still resolving.
6. "Show similar" (from a hotspot popover, a chip, or Shop-this-look's Replace button) opens `SimilarItemsPanel` — a clean product-card list to swap in an alternative, reusing the search that already ran rather than re-prompting for a link/photo.
7. Compare slider + download on the canvas. Shareable client link via `public_slug` — the share page (`/r/[id]`) mirrors the same immersive, tappable-hotspot treatment, read-only.

## Important gotchas
- `src/proxy.ts` (not `middleware.ts`) — Next.js 16 picks it up as middleware. Unauthenticated requests to protected routes get a 404 HTML page (Clerk's `protect-rewrite`), not a 401.
- Supabase free tier = 50 MB hard upload cap. **Do not use Supabase Storage for binary files.** DB only.
- Modal `@modal.web_endpoint` is deprecated — use `@modal.fastapi_endpoint`.
- The `supabase` pip package is no longer in the Modal worker (removed when switching to Vercel Blob).
- Vercel body size limit is 4.5 MB — all large file operations must go direct to Vercel Blob from the browser.
- **"SharedArrayBuffer is not allowed" on Vercel is an undici/fetch error, NOT a sharp error.** It's thrown by undici's fetch body validation (`allowShared:false`) when a Buffer whose `.buffer` is a genuine SharedArrayBuffer is used as a fetch body. Vercel Blob's `put()` sends the body via fetch, and on Vercel's Linux runtime `sharp`'s `toBuffer()` output is backed by a SharedArrayBuffer (libvips memory pool). It never reproduces on macOS. **Fix:** wrap any sharp-derived (or pool-backed) buffer in `toUnsharedBuffer()` (`src/lib/file-buf.ts`) before passing it to `put()`/fetch. Already applied at the single chokepoint `uploadImage()` in `src/lib/restyle-render.ts`. Do NOT chase this in sharp's *inputs* — that's the wrong layer. (`src/app/api/reels/route.ts` has the same latent pattern if it ever uploads a sharp-derived buffer.)
- Design system is **Warm Modern** (app-wide, not just restyle — replaced the earlier "Swiss Minimalism" system in a July 2026 reskin): `--primary: #1c1c1a` near-black (pills/CTAs), `--accent: #354733` forest green (buy/product actions — use the `accent`/`accentSoft` `Button` variants), warm off-white `--background: #faf9f6`, white `--card`. All buttons/chips/inputs/tabs are pill-shaped (`rounded-full`); cards/popovers/sheets use `rounded-2xl`/`rounded-3xl`. Depth comes from `shadow-[var(--shadow-soft)]` (resting) and `shadow-[var(--shadow-pop)]` (floating/hover) — `rounded-none` and `shadow-none` must not appear anywhere (grep-gated). 8px grid, tight `letter-spacing: -0.02em` headings. Tokens in `src/app/globals.css`. Use `lucide-react` icons, never emoji. Full primitive set lives in `src/app/(studio)/restyle/[id]/ui.tsx` (see the pages/components list above) and uses `cva`. `cn()` helper in `src/lib/utils.ts`. `shared.ts` (old rounded-corner style constants) was deleted along with the wizard — nothing in the restyle tree should import styling from anywhere but `ui.tsx` now. A hotspot pulse animation (`@keyframes hotspot-pulse` in globals.css) is disabled for everyone by the existing `prefers-reduced-motion` block — don't add a second guard around it; a `@keyframes fade-in` alongside it is used by `ProgressOverlay`'s rotating message.
- Detection is non-deterministic — always read the item list from the persisted `restyles.detected_objects`, never re-detect on each load (the chips would change every time).
- **Object detection is provider-pluggable (`src/lib/detect.ts`).** `detectObjects` dispatches to either Gemini (`detectObjectsGemini`, `gemini.ts`) or OpenAI vision (`detectObjectsOpenAI`, `openai.ts`) — BOTH use the shared `DETECT_PROMPT` and return the identical 0–1000 `box_2d` shape, so nothing downstream (hotspots, cropping, item list) cares which ran. Default is Gemini; set `DETECTION_PROVIDER=openai` (+ `OPENAI_API_KEY`) to switch. The dispatcher falls back to the other provider if the chosen one throws or returns nothing. For a same-image comparison without redeploying, `POST /api/restyle/detect` accepts `?provider=openai|gemini` (or `provider` in the JSON body) and overwrites `detected_objects`, so you can re-detect an existing room with each provider. **The primary accuracy lever is the Gemini TIER, not the provider:** research showed GPT-4o is not reliably better than Gemini at boxes, but `gemini-2.5-flash` is much weaker than the Pro tier — so `detectObjectsGemini` now runs on `GEMINI_DETECT_MODEL` (`gemini-2.5-pro`), which fixed the mis-boxing. The OpenAI path is kept only as an A/B, not the recommended default. Separately, the hotspot UI is now forgiving of imperfect boxes (highlighted regions, not single dots — see `ObjectHotspots.tsx`), so a slightly-off box no longer looks broken.
- **"Placed" UI (buy links, price) must never claim to be in the room when it isn't** — a past bug (`StagedStrip`, since deleted) showed full product cards for anything queued regardless of whether the displayed image reflected it, which reads as "the app is lying about what's in my room." `ChangesPanel` now satisfies this by HONEST LABELING rather than hiding: every relevant edit gets a persistent card (`ws.railEdits`, gated on the render's `signature`-derived `shownProductIds`), and its status chip — `in-room` / `pending — generate to apply` / `turning-off — generate to apply` / `off` — always tells the truth about whether it's actually pictured. Only `in-room` implies "this is really there right now." The public share page (`/r/[id]`) enforces the underlying rule server-side too: no hotspots at all when `current_url === original_url`.
- Deferred search cost note: this was a direct product decision (see the "Room Restyle flow" step 4) — don't reintroduce an automatic search-on-photo-upload without checking with the user first, since removing it was explicitly to avoid spending API/token cost before the user has committed to generating. It's now server-side (inside generate's `after()`) rather than a client loop, but the "only after generate, only for what's actually in the render" rule is unchanged.
- `next/server`'s `after()` is used three times in restyle: object detection at project-create time, Gemini scoring / Wayfair token resolution in `visual-search`, and (as of the fire-and-forget generate rework) the render itself plus the deferred inspo search in `generate/route.ts`. On Vercel, `after()` work runs within the route's own `maxDuration` (via `waitUntil`) — if you extend any of these background jobs, check the route's `maxDuration` covers it (`generate/route.ts` is `300`, sized for one render + several sequential product searches).
- **Generate is fire-and-forget, not a long synchronous request.** `POST /api/restyle/[id]/generate` returns `202` almost immediately (after applying any `{toggle}`/`{emptyRoom}` body and marking `restyles.generating_started_at`); the actual `recompose` + deferred search run in `after()`. The client (`useRestyleWorkspace`'s `pollGenerating`) is the ONLY place that adopts a finished render — same function whether it was just kicked off or resumed from a page reload that found `generating_started_at` already set. Don't add a second "read the response body for the render URL" path — the POST response has no `url`, only `{generatingStartedAt}`.
- **A per-object "remove from the room" is a `restyle_edits` row with `kind:"remove"` AND a `target_label`** — do not confuse this with the pre-existing "Empty the room" flag, which is `kind:"remove"` with a **null** `target_label` (whole-room, no specific item). Both share the same dedupe (`edits/route.ts` POST, `restyle-edits.ts`'s `stageEdit`): staging any of `item`/`add`/`remove` for a label deactivates the other two for that same label, since a slot can only hold one active outcome. A *placed* targeted remove gets no hotspot (`useRestyleWorkspace`'s `canvasHotspots` skips it — an empty patch of floor isn't tappable) and instead surfaces as a normal `ChangesPanel` card ("Removed the {label}") with the same on/off switch as any other change.
- **Custom free-text instructions for an item already in the room are `kind:"refine"` edits** — "mount the tv on the wall", "move the couch a bit to the left". Distinct dedupe group from `item`/`add`/`remove` (`edits/route.ts` POST): a refine is ADDITIVE to whatever else is happening in that slot (a swap can have its own refine on top), only one active refine per label at a time (a second instruction replaces, doesn't stack with, the first — avoids contradictory instructions piling up). `composeEdits`' `refine` case (`gemini.ts`) wraps the raw instruction with "make ONLY this specific change… keep everything else exactly as it is" so a vague free-text instruction can't accidentally rewrite the whole room. Entry point: `SourcePanel`'s standalone `"adjust"` view, reached from the "Adjust it" row of the item's edit MENU (see the reframed-editing gotcha) — `ws.stageRefine(label, instruction)`. Toggleable in `ChangesPanel` exactly like any other change (`ws.toggle` — a flag flip, see the batch-toggle gotcha), not a hard delete — turning it off keeps the instruction staged (just inactive) so re-enabling doesn't require retyping it. Deliberately NOT wired into `canvasHotspots`' per-object candidate matching (unlike item/remove) — a refine-only item shouldn't become the hotspot's "chosen" edit ahead of an actual product swap for the same label. Visibility in `ChangesPanel`: nested as a small annotation card on the product/inspo/remove card sharing that label (`refineFor` in `ChangesPanel.tsx`), or its own standalone card when the item has no other active/off edit.
- **Any existing item is EDITED, not just swapped** — tapping one (changed or not, detected item or a placed add) opens a category menu (`SourcePanel`'s `"menu"` view: Swap it / Find similar items / Adjust it / Remove it) rather than jumping straight into a sourcing form. This is why the placed/queued on-canvas popovers were removed: sourcing a replacement is only ONE of several things you can do, and a swapped item should have the exact same options as an un-swapped one (the user's explicit ask). Rows: `"Swap it"` → `view:"compose"` (link/photo/describe, `stagedEditId` carried so a pick replaces the current edit); `"Find similar items"` → `view:"similar"`; `"Adjust it"` → `view:"adjust"` (see custom-instructions gotcha); `"Remove it"` → `ws.stageRemove(label)` for a detected item, or `ws.remove(stagedEditId)` for a pure add (deleting the add edit — there's no original object to "remove"). A fresh "+ Add" (nothing placed yet) skips the menu — there's nothing to categorize — so it opens straight into `"compose"` with `hasMenu:false` (no "← Back").
- **Image sizing uses a JS-measured pixel box, not CSS `max-h-full` shrink-wrap — on every breakpoint, not just desktop.** The canvas stage now has a FIXED height everywhere (`h-[65dvh]` on mobile since the page scrolls below it, `flex-1` filling the immersive column on desktop), so a portrait photo needs to fit both axes while hotspots stay pixel-accurate. `max-height: 100%` on an image inside an auto-sized (shrink-wrapped) wrapper is genuinely ambiguous per the CSS spec — percentage heights don't resolve against a container whose own height comes from its content. `RestyleCanvas.tsx` (and `r/[id]/ShareCanvas.tsx`, which mirrors it, desktop-only for now) instead: watches the stage via `ResizeObserver`, computes the contained image's exact pixel box from the actual rendered `<img>`'s `naturalWidth`/`naturalHeight` (measured on load — preferred over trusting `restyle.width`/`height` alone, which is only the instant-first-paint fallback before the image loads), and pins the wrapper to that box via inline `style` (`position:absolute; left/top/width/height` in px). Hotspots/popovers keep their existing %-position code unchanged — they just land correctly because the wrapper IS the image's true rendered box, in pixels, not a CSS approximation. Falls back to plain `w-full h-auto` before the first measurement lands. Behind this sits a blurred, darkened `object-cover` copy of the same image (`blur-2xl brightness-50 scale-110`) filling the STAGE edge-to-edge (no rounding, no shadow, no border — it's just backdrop) — also now universal, not desktop-only. The sharp photo itself floats on top as a rounded, shadowed card (`rounded-3xl shadow-[var(--shadow-pop)] overflow-hidden` on `imgWrapClass`/`FALLBACK_WRAP`) — this is the RoomSketch-style look the user asked for after initially trying (and rejecting) a fully edge-to-edge un-rounded photo. On mobile, `RestyleStudio.tsx` renders the whole stage full-bleed ABOVE the padded column that holds `ChangesPanel`, not inside that padding.
- **The "+ Add" flow asks for location, then what it is, then how to source it — in that order.** `useRestyleWorkspace`'s `startAddFlow()` (triggered by the floating "+ Add" button) immediately shows `PinPlacementLayer` over the original photo via `pinRequest = {editId: null, label: ""}` — `editId: null` is the signal (to `RestyleCanvas` and to `PinPlacementLayer`'s handlers) that no edit exists yet, so placing/skipping a pin here (`placeAddLocation`/`skipAddLocation`) just stores `{x,y,note}` in `pendingAddPlacement` rather than PATCHing a real edit. Sourcing then opens with an empty label, and `SourcePanel` gates its normal tabs behind a one-line "What are you adding?" text step (`ws.setSourcingLabel`) whenever `mode==="add" && !label` — a fresh add-flow item always arrives here with an empty label, so this step always shows. Once the item is actually sourced (link/photo/description — all three tabs are available for "add" now, since a real label exists before you ever reach them) and the edit is created, `finalizeAddPlacement(editId, label)` attaches `pendingAddPlacement` immediately if one was captured, or falls back to the OLD post-hoc `requestPin` prompt if location was skipped — so a user who skips still gets one more chance right after the item lands. `pinRequest.editId: string` (non-null) is still what that post-hoc/legacy path uses, and is also how `setPlacement` (a real PATCH) is invoked from `RestyleCanvas`'s pin-layer branch — the two cases are told apart there by checking `ws.pinRequest.editId`.
- **"Find similar" works for ANY detected item, not just ones already swapped/staged.** `SimilarItemsPanel` used to only search off an already-staged edit's `reference_url` — a never-touched item had nothing to search from. `runVisualSearchByUrl` now takes an optional `box2d` (a detected object's own `box_2d`); when passed alongside the ORIGINAL photo's URL, `visual-search/route.ts` crops the original down to just that item server-side (`cropToBox`, same helper screenshot cleanup uses) before running the normal Lens+keyword pipeline — one shared code path either way. Entry point: `SourcePanel.tsx` shows a "Find similar items" button for any swap target with a detected match (`sourcing.mode==="swap" && matchedObject`), not just after staging something.
- A camera-captured **portrait** photo in `/restyle/new` can't be forced landscape by a web file input (no access to lens/zoom) — the capture screen shows a tip instead, and a portrait result triggers a retake prompt with an explicit "Use it anyway" escape hatch. Library uploads (existing photos) are never nagged — only the "Take a photo" path checks dimensions.
- `ProgressOverlay` (ui.tsx) is **self-ticking**: given `startedAt` + `expectedSeconds`, it runs its own 300ms interval internally to compute %, "~Xs left", and the rotating warm message index. This used to live in `useRestyleWorkspace` and re-render the ENTIRE editor tree every 300ms during a generate — same class of bug as the documented compare-slider fix (CLAUDE.md's original callout for that). If a similar "tick something during a long operation" need comes up again, keep the ticker local to the one component that displays it, not the shared hook.
