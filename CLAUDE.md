# Nook — Project Context for Claude

## What this app is
Nook is a SaaS for real estate agents. Three core features:
1. **3D Tours** — agent uploads a property walkthrough video → GPU processes it into a Gaussian Splat (.ply) → agent shares a 3D walkthrough link with clients
2. **Reels** — agent generates a cinematic 9:16 social media video from a property photo (via Higgsfield AI)
3. **Room Restyle** — agent uploads a room photo → taps a detected item on a full-screen, canvas-first editor → sources a replacement (a pasted retailer link, an inspo photo, or a description) or removes it from the room entirely → AI (Gemini "Nano Banana" image model) restages the room → each swapped item is matched to a real, buyable product ("shop the look") via Google Lens + retailer APIs, searched only AFTER generate (not the moment a photo is uploaded) → agent shares a client link with the same tappable, shop-the-look experience. Generate is fire-and-forget end to end (see below) — the whole thing runs server-side, so nothing is lost if the tab closes. This is the feature under active development.

Live app: https://nook-lime.vercel.app  
GitHub: https://github.com/HyperboleDigital/nook

---

## Stack
- **Framework**: Next.js 16 (App Router, `src/` dir, TypeScript)
- **Auth**: Clerk (`src/proxy.ts` is the middleware — Next.js 16 picks this up, NOT `middleware.ts`)
- **Database**: Supabase (DB only — no Supabase Storage, tables: `users`, `tours`, `reels`)
- **File storage**: Vercel Blob (videos up to 2 GB, PLY output files) — `BLOB_READ_WRITE_TOKEN` required
- **GPU processing**: Modal.com serverless GPU (`modal/worker.py`) — deployed as `nook-3dgs`
- **3D pipeline**: Nerfstudio `splatfacto` on Modal A10 GPU (~35 min, ~$0.73/scan)
- **3D viewer**: SuperSplat (`superspl.at/editor?load=<ply_url>`) — display only, not a processor
- **Reels**: Higgsfield AI API
- **Room Restyle**: Gemini image model (`gemini-2.5-flash-image` "Nano Banana", pro variant `gemini-3-pro-image-preview`) for compositing; `sharp` for image canonicalization; Google Lens + SerpApi / Unwrangle / Rainforest for product matching
- **Billing**: Stripe (plans: free / starter / pro)

All three pipelines are wired and deployed. **Room Restyle is the feature under active development** (most recent work). Storage is Vercel Blob throughout (Supabase free tier hard-caps at 50 MB; property videos are 200 MB–2 GB).

**What works:**
- Clerk auth with route protection via `src/proxy.ts`

- Room Restyle: **immersive, full-viewport editor** living in its own `(studio)` route group (no app sidebar/padding — see "Room Restyle — pages/components" below), room photo edge-to-edge with a floating rail on desktop, tap a hotspot to source/remove an item, server-persisted product search deferred until after generate, "Similar items" + "Shop this look" + "Queued changes" panels, a fire-and-forget Generate with a self-ticking progress overlay (% + rotating warm messages), a per-object "remove from room" action distinct from un-toggling a swap, and a shareable client link with the same tappable hotspots (read-only). Exercise the real flow before assuming it's fully correct, especially the desktop measured-image-box sizing (see gotchas) and the fire-and-forget generate/poll cycle.

**Pending / known issues:**
- Clerk keys are still `pk_test_` (dev mode) — needs production keys before real users
- Supabase Storage bucket `nook-uploads` still exists but is no longer used
- No usage/quota enforcement on free plan yet
- **`supabase/migrations/013_restyle_searches.sql` needs to be applied manually** (no Supabase CLI is configured in this project — migrations are pasted into the Supabase SQL editor by hand). Until it's run, `GET /api/restyle/[id]/searches` 500s harmlessly (the client swallows the error and just starts with an empty search cache) — search/pick still works, results just won't persist across reloads.
- **`supabase/migrations/014_restyle_room_type.sql` also needs manual application.** Adds a nullable `room_type` column to `restyles`, set by the capture wizard's room-type picker (`/restyle/new`). The insert in `POST /api/restyle` only includes `room_type` when a value was actually picked, so uploads work fine even before this migration is applied — the field is just silently dropped until then.
- **`supabase/migrations/015_restyle_placement.sql` also needs manual application — BEFORE deploying the pin-placement code.** Adds `restyle_edits.placement` jsonb (`{x, y, note}`, 0–1000 box_2d space, set by the add-item tap-to-place pin) and drops the dead `restyle_versions` table. Unlike 014, this one is not optional-at-runtime: `PATCH /api/restyle/[id]/edits` writes the `placement` column directly and will 500 on an unknown column.
- **`supabase/migrations/016_restyle_generating_status.sql` also needs manual application — BEFORE deploying the resumable-generate code.** Adds `restyles.generating_started_at`/`generate_error`, set/cleared by `POST /api/restyle/[id]/generate` so a fresh page load can detect an in-flight render and resume showing progress instead of it finishing invisibly. Not optional-at-runtime: the generate route writes these columns directly and will 500 on an unknown column.
- Restyle hotspots on a *render* are an approximation, not real detection: object positions are only ever detected on the original photo (`restyles.detected_objects`), so a hotspot on the styled result reuses that item's ORIGINAL box_2d position. Correct for the common case (a swap usually stays roughly where the original piece was) but can drift if Nano Banana repositions/resizes furniture significantly. A pinned "add" item uses a small box synthesized around its tap-to-place pin instead; an unpinned add gets no hotspot and shows only in "Shop this look".
- **The render is a canvas, not a dead end.** `useRestyleWorkspace`'s `canvasHotspots` puts a hotspot on EVERY detected item on whichever image is displayed (original or a render), each in one of three states: `idle` (unchanged — tap to swap), `queued` (staged but not in the pictured image — tap for the `QueuedHotspotPopover` teaser), `placed` (the change IS in the pictured image — tap for the priced `HotspotPopover`). `placed` is derived from the displayed render's `signature` (`shownProductIds`), which is always empty while `viewingOriginal` — so the "never show placed/priced UI on the original" rule (below) holds automatically instead of needing a second code path. This means a user can keep tapping unchanged items on a render to keep swapping/adding and regenerating — `recompose` always composites fresh from the ORIGINAL photo plus the current active-edit set, so iterating from a render needed no backend change, only this hotspot model. There's no chip row anymore — tapping the photo directly (via `canvasHotspots`) is the only sourcing entry point; a floating "+ Add" button lives on the canvas itself (`RestyleCanvas.tsx`) for new items.
- **Hotspot/pin popovers must clamp their horizontal position in PIXELS, not just percent of the image.** A percent-only clamp (e.g. `min(max(cx, 18), 82)`) assumes the rendered image is wide enough that half the popover's fixed width is a small percentage of it — that assumption breaks on a narrower image (portrait photo, smaller viewport, a side-by-side desktop layout) and the fixed-width card clips off the edge of the `overflow-hidden` canvas frame. Fix pattern (see `HotspotPopover.tsx`, `QueuedHotspotPopover.tsx`, `PinPlacementLayer.tsx`): `left: clamp(HALF_WIDTH_PX, ${cx}%, calc(100% - HALF_WIDTH_PX))` — CSS `clamp()` correctly compares mixed px/% units, so the card's edge never gets closer than `HALF_WIDTH_PX` to either side of the frame regardless of its actual width.
- **Don't put drag-driven state in `useRestyleWorkspace` — it re-renders the entire editor tree.** The compare slider used to live in the shared hook and called `setCompare` on every raw `pointermove`, which re-rendered ShopLook/QueuedChanges/GenerateBar/etc. on every pixel of drag (the hook backs the whole `RestyleStudio` subtree, not just the canvas), which is what made it feel laggy. Fixed by moving `compare`/`dragging`/`imgWrapRef`/`sliderHandlers` into local state in `RestyleCanvas.tsx` (nothing else read them) and throttling `setCompare` to at most once per animation frame via a `requestAnimationFrame` guard. If something is only read by one component, keep its state there — don't default to putting it in the shared hook just because everything else lives there.

### Room Restyle — API routes (`src/app/api/restyle/`)
- `upload-url/route.ts` — POST, signed Vercel Blob upload token for room/inspo photos (`handleUpload`, mirrors `tours/upload-url/route.ts`). The browser uploads the actual image bytes straight to Blob using `@vercel/blob/client`'s `upload()`, decoupled from our own functions — closing the tab mid-transfer just loses that unfinished upload, never a half-created DB row. `route.ts` and `[id]/product/route.ts` below only ever receive the resulting blob URL as JSON.
- `route.ts` — POST, create a restyle project from an already-uploaded room photo (`{photoUrl, title?, room_type?}` JSON — no file bytes in this request). Fetches the blob, `sharp` canonicalizes (EXIF-rotate + downscale to 1536px), re-uploads the canonical copy, best-effort deletes the client's raw upload, inserts `restyles` row, then fires object detection in the background via Next's `after()` using the buffer already in memory (no refetch) — chips are usually ready by the time the editor loads instead of a separate client-triggered detect call.
- `detect/route.ts` — legacy/fallback path: Gemini detects editable objects in the room (JSON `{imageUrl,restyleId}` or multipart). The client only calls this if the background detection from `route.ts` hasn't landed after ~20s of polling. Caches `detected_objects` on the project (detection is non-deterministic, so it's persisted for a stable item list).
- `[id]/product/route.ts` — POST, stage a product/photo as a reference edit. Input shapes, all JSON: `{url}` (pasted retailer link — a confirmed real product), `{token}` (SerpApi immersive token → resolved URL), or `{imageUrl}` (an inspo photo the client already uploaded to Blob via `upload-url/route.ts` — staged as a reference with **no buy link yet**; search is deferred, see below). `targetLabel` force-targets a specific slot; `replaceEditId` deletes a superseded edit (e.g. an inspo photo now replaced by a real pick) in the same request. Auto-decides replace-vs-add via `matchDetected`. Does NOT render. Shared staging logic (insert + single-active-per-label dedupe) lives in `src/lib/restyle-edits.ts`'s `stageEdit()`.
- `[id]/visual-search/route.ts` — POST, find buyable products via Google Lens + keyword search, Gemini-scored against the target image. Three input shapes: `image` (file), `imageUrl` (an already-staged/cropped photo — used for the deferred post-generate search, see below), or `query` (text, from "Describe it"). Responds fast with unscored results, then finishes Gemini scoring + resolves Wayfair immersive tokens in the background via `after()`, persisting the final set to `restyle_searches` (see DB tables). Does NOT render.
- `[id]/searches/route.ts` — GET, the persisted search results for a project (optional `?label=`). Replaces what used to be a client-side localStorage cache.
- `[id]/edits/route.ts` — POST add / PATCH toggle / DELETE a change layer (multipart). Does NOT render. POST's single-active-per-`target_label` dedupe covers `item`/`add`/`remove` together (a swap, an add, and a targeted remove of the same slot are mutually exclusive — staging any one deactivates the others). PATCH also accepts a batch `{ states: {editId: bool} }` map for atomically flipping several edits in one call (used by "Start from original").
- `[id]/generate/route.ts` — POST, **fire-and-forget**: applies an optional server-side edit-state change (`{toggle: {editId,active}}` or `{emptyRoom: true}` — atomic, replaces what used to be a client for-loop that died if the tab closed mid-sequence), marks `generating_started_at`, and responds `202` immediately. The actual render (`recompose`) **and** the deferred inspo product search (via `src/lib/restyle-search.ts`) both run inside `after()` — so the whole "make the image, then find buyable matches for anything that was just an inspo photo" sequence survives the client disconnecting entirely. `maxDuration = 300` (covers render + N searches). The client (`useRestyleWorkspace`'s `pollGenerating`) polls `GET /api/restyles/[id]` until `generating_started_at` clears.
- `[id]/items/route.ts` — item list helper (custom/not-detected items).
- `restyles/route.ts`, `restyles/[id]/route.ts` — list/read restyle projects (plural — the gallery/index endpoints). `GET /api/restyles/[id]` returns `{...restyle, edits, renders}` and is the single read both the initial load and generate-polling use.

### Room Restyle — pages/components
The editor (`[id]/`) lives in its own **`src/app/(studio)/restyle/[id]/`** route group — a chrome-free, full-viewport shell (see `(studio)/layout.tsx`) with no app sidebar/padding/max-width, so the room photo can go edge-to-edge. The gallery and capture wizard stay in the normal app shell: `src/app/(app)/restyle/page.tsx` (projects gallery) and `src/app/(app)/restyle/new/page.tsx` (upload/paste a room photo — clipboard paste supported; "Take a photo" only on phones, "Choose a photo" on desktop; client-side `downscaleImage()` before upload, see `src/lib/image-client.ts`; a camera-captured portrait photo triggers a "Landscape works best" retake prompt with a "Use it anyway" escape hatch — a web file-input camera can't force landscape orientation or the 0.5x lens, so this is a post-hoc check, not a real constraint). The URL is unchanged (`/restyle/[id]`) — route groups don't affect routing, and Clerk protection in `src/proxy.ts` is path-based so auth is unaffected by the move.
- `(studio)/restyle/[id]/page.tsx` — thin shell: `h-dvh flex flex-col` with a slim `h-12` top bar (back arrow, title input) + `<RestyleStudio>` filling the rest of the viewport.
- `(studio)/restyle/[id]/useRestyleWorkspace.ts` — the single state hook everything else reads from. Key concepts: `sourcing` (the item currently being sourced — `{label, mode: "swap"|"add", view: "compose"|"similar", stagedEditId, lastStaged?}`, cleared entirely on panel close so stale banners can't outlive the item they describe), `searches` (per-label search state hydrated from `GET /searches` and updated in place as scoring finishes), `canvasHotspots` (unified hotspot list for whichever image is on screen — see gotchas), `viewingOriginal`/`stagedItems`/`productEdits`/`inspoEdits`/`removedEdits` (derived — see gotchas for what each one filters by). Picking a candidate (`pickCandidate`) is optimistic: the edit appears staged immediately using the search result's own thumbnail/title/price, reconciled or rolled back once the server responds. `toggle` flips one edit's `active` flag alone (no render, still used standalone nowhere but kept for clarity); `toggleAndRegenerate` (per-item on/off in `HotspotPopover`/`ShopLook`) does an optimistic local flip then calls `generate({toggle})` so the server applies the same flip atomically as part of the same fire-and-forget render — a previously-seen combination is a free cache hit via `restyle_renders`' signature cache, a new one pays a real render. `deactivateAll()` batch-PATCHes every active edit off in one call (used by "Start from original" — replaces a former client for-loop). `stageRemove(label)` stages a targeted `kind:"remove"` edit for a detected item (see "per-object remove" gotcha below). The before/after compare slider's state (`compare`/`imgWrapRef`/`sliderHandlers`) is NOT in this hook — it lives locally in `RestyleCanvas.tsx` (see gotchas) since nothing else needs it. `generate(body?)` is fire-and-forget: POST returns 202 immediately, then `pollGenerating` (also used to resume an in-flight render found on page load, via `restyles.generating_started_at` — migration 016) polls `GET /api/restyles/[id]` every ~4s until it clears, adopts the fresh `current_url`/`edits`/`renders`, surfaces `generate_error`, and re-fetches `/searches` once to hydrate whatever the server-side deferred search produced while nobody was watching. `generating` is derived from `generatingStartedAt !== null`; `expectedSeconds` (`clamp(30 + 8×activeEdits, 30, 90)`) and `generatingStartedAt` are handed to `ProgressOverlay`, which does its own ticking (see gotchas — this used to tick in the hook and re-render the whole tree every 300ms).
- `(studio)/restyle/[id]/RestyleStudio.tsx` — layout owner, branches fully by breakpoint (not just CSS tweaks) since desktop and mobile have genuinely different structures. Desktop: `RestyleCanvas` fills an immersive `relative h-full w-full` stage; a **floating, collapsible right rail** (`absolute right-4 top-4 bottom-4 w-[380px]`, rounded card, shadow-pop) defaults to `QueuedChanges` (viewing the original) or `ShopLook` (viewing a render) and swaps to `SourcePanel`/`SimilarItemsPanel` while `sourcing` is open; `GenerateBar` floats bottom-center as a pill (`variant="floating"`); `VersionsStrip` floats bottom-left in a small card. Mobile: the previous stacked-column layout (scroll down through canvas → Queued/Shop → versions → sticky `GenerateBar`), sourcing in a bottom `Sheet`.
- `(studio)/restyle/[id]/RestyleCanvas.tsx` — the room photo, treated as a live canvas whether it's the original or a render (see the `canvasHotspots` gotcha — a render is not a dead end, every item stays tappable). The stage has a FIXED height on every breakpoint now (mobile and desktop both use the measured-pixel-box + blurred-backdrop treatment — see the "image sizing" gotcha for why CSS shrink-wrap alone doesn't work), edge-to-edge with no border/rounded corners anywhere. Renders `ObjectHotspots` + whichever popover matches a tapped hotspot's state (`QueuedHotspotPopover` for `queued`, `HotspotPopover` for `placed` — thumbnail/price/Show-similar/Buy), compare slider, share/download.
- `(studio)/restyle/[id]/ObjectHotspots.tsx` — circular tap targets from `box_2d`, positioned as percentages, styled per-hotspot state (idle/confirming/queued/placed — see gotchas). Solid white backing disc under every idle marker (a plain small dot disappears against a photo). Two placement passes: (1) `anchorFor` — a compact item pins at its box center, but a SURFACE item (rug, big sofa; box ≥ `SURFACE_AREA_FRAC` of the image) pins at the point in its box closest to center that isn't covered by a SMALLER box, so a rug pins on visible rug instead of under the coffee table sitting on it (only smaller boxes count as occluders, so a sofa still pins on the sofa despite overlapping the larger rug beneath it); (2) `declutter` — a pairwise separation pass so hotspots detected close together (e.g. a vase on a coffee table) don't overlap into one unreachable blob.
- `(studio)/restyle/[id]/SourcePanel.tsx` — the link/photo/describe sourcing form, for an EMPTY slot only. Shows a `CroppedThumb` of the actual item being replaced next to the label. Photo staging is optimistic (see `stagePhoto` gotcha). For a swap slot with a real detected match, also offers "Remove this {label} from the room" (`Eraser` icon, ghost style) at the bottom — calls `ws.stageRemove` (see the per-object remove gotcha).
- `(studio)/restyle/[id]/SimilarItemsPanel.tsx` — clean product-card list ("Recommended based on X") for a slot that already has something placed — the "Show similar"/"Find buyable matches" destination. No tabs; auto-triggers a search against the staged item's own reference photo if one hasn't run yet.
- `(studio)/restyle/[id]/CroppedThumb.tsx` — crops a region out of a full photo using CSS position/size math only (no canvas/server round-trip) — used to show "here's the actual item" next to a label instead of just text.
- `(studio)/restyle/[id]/QueuedChanges.tsx` — right-rail default while viewing the original: pending swaps/adds/removes, explicitly framed as "not in the photo yet" (do not conflate with `ShopLook` — see gotchas, this distinction fixed a real bug). A queued `remove` renders with an `Eraser` icon and "Removing" label instead of a thumbnail. Collapsed by default behind an "n changes queued" toggle — each item already has a green-checkmark hotspot on the canvas, so this list is a secondary "review everything at once" view, not the primary way to see what's queued.
- `(studio)/restyle/[id]/QueuedHotspotPopover.tsx` — teaser anchored to a `queued` hotspot (thumbnail, "Queued" badge, Change/Remove) — the on-canvas counterpart to `QueuedChanges`, same "not in the photo yet" framing, never price/Buy. For `kind:"remove"` it shows "Will be removed" framing (no thumbnail) with a single "Keep it — don't remove" action instead of Change/Remove.
- `(studio)/restyle/[id]/ShopLook.tsx` — right-rail default while viewing a render: exactly what's actually shoppable in the CURRENT image. Real products get a full card (retailer/price/Buy/Replace); inspo-only items get a compact "from your photo" card with a "Find buyable matches" button that opens `SimilarItemsPanel`; a **"Removed" section** lists any item actually removed from this render with a Restore button (`toggleAndRegenerate(id, false)` — a free cache hit if that exact combination was rendered before).
- `(studio)/restyle/[id]/GenerateBar.tsx` — primary Generate + overflow menu (⋯) for "Empty the room" / "Start from original". `variant="floating"` (desktop, self-contained rounded pill) vs `"sticky"` (mobile, default, spans the bottom of the stacked column).
- `(studio)/restyle/[id]/VersionsStrip.tsx` — render-history thumbnails.
- `(studio)/restyle/[id]/ui.tsx` — Warm Modern primitives (cva + lucide-react): `Button` (variants include `accent`/`accentSoft` for buy/product actions), `IconButton`, `ProductCard`, `Sheet`/`SheetChrome` (mobile bottom-sheet; desktop docking is custom-built in `RestyleStudio`, not `Sheet`'s job anymore), `Chip`, `Spinner`, `Input`, `StatusBanner`, `Skeleton`/`SkeletonProductCard`, `ProgressOverlay` (self-ticking — see gotchas), `SegmentedTabs`, `SectionLabel`, `shopSummary`/`ShopSummaryPill` (the "n items · from $X" total, shared by `ShopLook` and the canvas's floating pill), `storeName` (retailer display name from a URL — also reused by the public share page).

### Room Restyle — libraries (`src/lib/`)
- `gemini.ts` — Gemini client. Every call now has a real fetch timeout (`AbortSignal.timeout`, 20s vision / 90s image-gen) — previously unbounded, relying only on the route's `maxDuration`. Key exports: `restyleRoom`, `composeEdits` (multi-edit compositor — the `remove` case emits a *targeted* removal prompt when `targetLabel` is set, "remove all furniture" only when it's a label-less whole-room remove), `detectObjects`, `describeProduct`/`describeProductImages` (recover dimensions/proportions for accurate scale), `describeScreenshotForSearch` (also extracts a literal product title/brand if legible text is visible — stronger search signal than a generic description for niche/branded items), `locateProductPhoto` (crops UI chrome out of a screenshot before it pollutes Lens matching), `scoreImageMatches`. Models: `GEMINI_IMAGE_MODEL` = `gemini-2.5-flash-image`, `GEMINI_IMAGE_PRO_MODEL` = `gemini-3-pro-image-preview`, `GEMINI_VISION_MODEL` = `gemini-2.5-flash`.
- `restyle-edits.ts` — shared "stage a reference edit" logic (`loadOwnedRestyle`, `editsFor`, `matchDetected`, `stageEdit`) used by both the product route and (historically) visual-search's stage mode, so the insert+dedupe logic doesn't drift between callers. `stageEdit`'s dedupe also covers `remove` alongside `item`/`add`.
- `restyle-search.ts` — `searchProductByImageUrl({restyleId, imageUrl, label})`, extracted so the SAME product-search pipeline (Lens + keyword search → fast unscored results, persisted → `finish()` continuation that does Gemini scoring + Wayfair token resolution → persisted final set) can be called both from `visual-search/route.ts`'s `imageUrl` path AND from the generate route's `after()` for the deferred inspo search — one implementation, two callers, instead of duplicating the pipeline. Also exports `upsertSearch`.
- `restyle-render.ts` — `uploadImage` (Blob upload chokepoint — see SharedArrayBuffer gotcha), `recompose` (caches renders per active-edit signature in `restyle_renders`; original + every reference photo fetch in parallel — deliberately a PURE "render" function with no lifecycle/status bookkeeping; that lives at the generate route's boundary), `closestAspect`.
- `product.ts` — retailer product detail (Unwrangle for Wayfair/Walmart/Home Depot, Rainforest for Amazon).
- `shopping-search.ts` — SerpApi Google Lens visual match + Google Shopping/Amazon/Walmart/HD keyword search (always run in parallel, not gated behind a match-count threshold); `resolveImmersiveToken` resolves ONE Wayfair immersive token → a real URL. **Token resolution is now LAZY (a SerpApi-cost decision):** `resolveTokenUrls` (which resolved every token-only candidate in the background) is no longer wired into the search flow — the product route resolves a token only when that specific candidate is actually picked (`product/route.ts`'s `{token}` branch). This cut SerpApi usage substantially (most candidates are never picked); the only tradeoff is a token-only Wayfair card shows no pre-resolved "View on X" link until picked. Don't re-add eager `resolveTokenUrls` without a SerpApi-budget reason. Results are filtered to `supported` only before ever reaching the client — an unshoppable card is just noise.
- `image-crop.ts` — `cropToBox` (sharp-based crop from a Gemini box_2d, used server-side for screenshot cleanup).
- `image-client.ts` — `downscaleImage` (browser-side canvas resize before upload — phone photos routinely exceed Vercel's 4.5MB body limit).
- `file-buf.ts` — `fileToBuffer` (stream-read a File/Blob) and `toUnsharedBuffer` (strip SharedArrayBuffer backing before a fetch/Blob-put body — see gotcha).

### Room Restyle — public share page
- `src/app/r/[id]/page.tsx` — server component, no auth (`/r/(.*)` is public in `src/proxy.ts`). Computes the same hotspot list the editor would (placed product edits ↔ `detected_objects` boxes, or a synthesized box around an "add" edit's pin), passed as plain props to a client component. Hotspots are empty whenever `current_url === original_url` (nothing's ever been generated) — same "never show placed UI on the unedited photo" rule as the editor.
- `src/app/r/[id]/ShareCanvas.tsx` — client component, same full-viewport immersive treatment and measured-image-box technique as `RestyleCanvas` (see gotchas), but read-only: tapping a hotspot opens a small thumbnail/price/Buy popover only — no Show similar, no toggle, no sourcing. Desktop: floating "Shop this look" panel on the right. Mobile: stacked column with inline hotspots on the plain `w-full h-auto` image.

### Restyle DB tables (Supabase)
- `restyles` — project (original_url, current_url, width/height, detected_objects, custom_items, title, user_id, public_slug, generating_started_at/generate_error — migration 016)
- `restyle_edits` — change layers (kind `item`/`add`/`style`/`remove`/`refine`, target_label, reference_url/desc, buy_url, product_title/price, active, position). A `remove` edit with a `target_label` removes just that one detected item (see the per-object remove gotcha); a `remove` edit with a null `target_label` is the whole-room "Empty the room" flag.
- `restyle_renders` — cached renders keyed by `signature` (comma-joined active edit ids) for instant toggle-back
- `restyle_searches` (013_restyle_searches.sql, **needs manual application** — see "Pending / known issues") — persisted product-search results per `(restyle_id, label)`: `query`, `results` (jsonb `ShoppingResult[]`), `scored` (bool — false until Gemini scoring + Wayfair token resolution finish in the background). Replaces the old client-side localStorage cache.

## Room Restyle flow (end-to-end)
1. Agent goes to `/restyle/new`, uploads or pastes a room photo → browser uploads it directly to Vercel Blob (`POST /api/restyle/upload-url` for the signed token, then `@vercel/blob/client`'s `upload()`) → `POST /api/restyle` `{photoUrl}` (sharp canonicalizes, stores original, creates project, fires detection in the background) → redirect to `/restyle/[id]`. Splitting the byte transfer from project creation means closing the tab mid-upload just loses the unfinished Blob transfer, never a half-created project.
2. Immersive full-viewport editor loads (`(studio)/restyle/[id]`): room photo edge-to-edge with tappable hotspot circles (real detected positions). Right rail (floating on desktop, inline on mobile) shows "Queued changes" (empty at first).
3. Agent taps a hotspot. Sourcing panel opens (floating rail on desktop, bottom sheet on mobile):
   - **Paste a link** → `POST /api/restyle/[id]/product` `{url}` stages it immediately as a confirmed real product (buy link known right away, no search needed).
   - **Upload a photo** → same direct-to-Blob upload as step 1, then `POST /api/restyle/[id]/product` `{imageUrl}` stages it as **inspo only** — no shopping search runs yet. This is deliberate: search used to fire the instant a photo was picked, spending API/token cost before the user had even seen the room or decided the item was worth shopping for.
   - **Describe it** → manual "Find" triggers `POST /api/restyle/[id]/visual-search` (text `query`).
   - **Remove it** → `ws.stageRemove(label)` stages a targeted `kind:"remove"` edit for that detected item — a distinct action from un-toggling a swap (which just reverts to the original item); this actually erases it from the room.
4. Agent hits Generate → `POST /api/restyle/[id]/generate` responds `202` immediately (fire-and-forget); server-side in `after()`: `recompose` composites all active edits with Gemini and uploads the render (cached per active-edit signature), **then the deferred search happens**: for every active edit that's inspo-only (a reference photo, no buy_url), `runProductSearch`-style lookup (via `restyle-search.ts`) finds real buyable matches — scoped to whatever actually made it into the render. The client polls until this all finishes, showing a self-ticking progress overlay (% + rotating warm messages + estimated seconds) the whole time — closing the tab mid-generate loses none of this, it all completes server-side and rehydrates on the next load.
5. Canvas now shows the render. Hotspots reappear (positions approximated from each swapped item's ORIGINAL detected box — see gotchas; a placed *remove* shows no hotspot at all) with a popover (thumbnail/price/Show-similar/Buy) on tap. Right rail switches to "Shop this look": shoppable items as cards (retailer/price/buy link/running total), a "Removed" section with Restore for anything erased from the room, plus inline search results for anything still resolving.
6. "Show similar" (from a hotspot popover, a chip, or Shop-this-look's Replace button) opens `SimilarItemsPanel` — a clean product-card list to swap in an alternative, reusing the search that already ran rather than re-prompting for a link/photo.
7. Compare slider + download on the canvas. Shareable client link via `public_slug` — the share page (`/r/[id]`) mirrors the same immersive, tappable-hotspot treatment, read-only.

## Important gotchas
- `src/proxy.ts` (not `middleware.ts`) — Next.js 16 picks it up as middleware. Unauthenticated requests to protected routes get a 404 HTML page (Clerk's `protect-rewrite`), not a 401.
- Supabase free tier = 50 MB hard upload cap. **Do not use Supabase Storage for binary files.** DB only.
- Modal `@modal.web_endpoint` is deprecated — use `@modal.fastapi_endpoint`.
- The `supabase` pip package is no longer in the Modal worker (removed when switching to Vercel Blob).
- Vercel body size limit is 4.5 MB — all large file operations must go direct to Vercel Blob from the browser.
- **"SharedArrayBuffer is not allowed" on Vercel is an undici/fetch error, NOT a sharp error.** It's thrown by undici's fetch body validation (`allowShared:false`) when a Buffer whose `.buffer` is a genuine SharedArrayBuffer is used as a fetch body. Vercel Blob's `put()` sends the body via fetch, and on Vercel's Linux runtime `sharp`'s `toBuffer()` output is backed by a SharedArrayBuffer (libvips memory pool). It never reproduces on macOS. **Fix:** wrap any sharp-derived (or pool-backed) buffer in `toUnsharedBuffer()` (`src/lib/file-buf.ts`) before passing it to `put()`/fetch. Already applied at the single chokepoint `uploadImage()` in `src/lib/restyle-render.ts`. Do NOT chase this in sharp's *inputs* — that's the wrong layer. (`src/app/api/reels/route.ts` has the same latent pattern if it ever uploads a sharp-derived buffer.)
- Design system is **Warm Modern** (app-wide, not just restyle — replaced the earlier "Swiss Minimalism" system in a July 2026 reskin): `--primary: #1c1c1a` near-black (pills/CTAs), `--accent: #354733` forest green (buy/product actions — use the `accent`/`accentSoft` `Button` variants), warm off-white `--background: #faf9f6`, white `--card`. All buttons/chips/inputs/tabs are pill-shaped (`rounded-full`); cards/popovers/sheets use `rounded-2xl`/`rounded-3xl`. Depth comes from `shadow-[var(--shadow-soft)]` (resting) and `shadow-[var(--shadow-pop)]` (floating/hover) — `rounded-none` and `shadow-none` must not appear anywhere (grep-gated). 8px grid, tight `letter-spacing: -0.02em` headings. Tokens in `src/app/globals.css`. Use `lucide-react` icons, never emoji. Full primitive set lives in `src/app/(studio)/restyle/[id]/ui.tsx` (see the pages/components list above) and uses `cva`. `cn()` helper in `src/lib/utils.ts`. `shared.ts` (old rounded-corner style constants) was deleted along with the wizard — nothing in the restyle tree should import styling from anywhere but `ui.tsx` now. A hotspot pulse animation (`@keyframes hotspot-pulse` in globals.css) is disabled for everyone by the existing `prefers-reduced-motion` block — don't add a second guard around it; a `@keyframes fade-in` alongside it is used by `ProgressOverlay`'s rotating message.
- Detection is non-deterministic — always read the item list from the persisted `restyles.detected_objects`, never re-detect on each load (the chips would change every time).
- **Object detection is provider-pluggable (`src/lib/detect.ts`).** `detectObjects` dispatches to either Gemini (`detectObjectsGemini`, `gemini.ts`) or OpenAI vision (`detectObjectsOpenAI`, `openai.ts`) — BOTH use the shared `DETECT_PROMPT` and return the identical 0–1000 `box_2d` shape, so nothing downstream (hotspots, cropping, item list) cares which ran. Default is Gemini; set `DETECTION_PROVIDER=openai` (+ `OPENAI_API_KEY`) to switch. Built as an A/B because Gemini sometimes mis-boxes/mislabels items (a "refrigerator" box landing on a sofa pillow was the trigger). The dispatcher falls back to the other provider if the chosen one throws or returns nothing. For a same-image comparison without redeploying, `POST /api/restyle/detect` accepts `?provider=openai|gemini` (or `provider` in the JSON body) and overwrites `detected_objects`, so you can re-detect an existing room with each provider. Note: placement math (`ObjectHotspots`' `anchorFor`) only fixes where a pin sits GIVEN a correct box — a wrong box is a detection problem, which is what this switch addresses.
- **"Placed" UI (the hotspot popover, buy links, price) must never show on the ORIGINAL photo** — only on a render, where the item is genuinely visible. A past bug (`StagedStrip`, since deleted) showed full product cards for anything queued to generate regardless of whether the currently-displayed image reflected it, which reads as "the app is lying about what's in my room." The fix pattern: gate on `ws.viewingOriginal` (or the render's `signature`-derived `shownProductIds`, which `productEdits`/`inspoEdits`/`canvasHotspots` all filter through) — never on "is something staged" alone. `QueuedChanges` (right rail) and `QueuedHotspotPopover` (on-canvas) are the deliberately-different "pending, not real yet" views for while `viewingOriginal` is true (or a change is staged but not yet in the displayed render). The public share page (`/r/[id]`) enforces the same rule server-side: no hotspots at all when `current_url === original_url`.
- Deferred search cost note: this was a direct product decision (see the "Room Restyle flow" step 4) — don't reintroduce an automatic search-on-photo-upload without checking with the user first, since removing it was explicitly to avoid spending API/token cost before the user has committed to generating. It's now server-side (inside generate's `after()`) rather than a client loop, but the "only after generate, only for what's actually in the render" rule is unchanged.
- `next/server`'s `after()` is used three times in restyle: object detection at project-create time, Gemini scoring / Wayfair token resolution in `visual-search`, and (as of the fire-and-forget generate rework) the render itself plus the deferred inspo search in `generate/route.ts`. On Vercel, `after()` work runs within the route's own `maxDuration` (via `waitUntil`) — if you extend any of these background jobs, check the route's `maxDuration` covers it (`generate/route.ts` is `300`, sized for one render + several sequential product searches).
- **Generate is fire-and-forget, not a long synchronous request.** `POST /api/restyle/[id]/generate` returns `202` almost immediately (after applying any `{toggle}`/`{emptyRoom}` body and marking `restyles.generating_started_at`); the actual `recompose` + deferred search run in `after()`. The client (`useRestyleWorkspace`'s `pollGenerating`) is the ONLY place that adopts a finished render — same function whether it was just kicked off or resumed from a page reload that found `generating_started_at` already set. Don't add a second "read the response body for the render URL" path — the POST response has no `url`, only `{generatingStartedAt}`.
- **A per-object "remove from the room" is a `restyle_edits` row with `kind:"remove"` AND a `target_label`** — do not confuse this with the pre-existing "Empty the room" flag, which is `kind:"remove"` with a **null** `target_label` (whole-room, no specific item). Both share the same dedupe (`edits/route.ts` POST, `restyle-edits.ts`'s `stageEdit`): staging any of `item`/`add`/`remove` for a label deactivates the other two for that same label, since a slot can only hold one active outcome. A *placed* targeted remove gets no hotspot (`useRestyleWorkspace`'s `canvasHotspots` skips it — an empty patch of floor isn't tappable) and instead surfaces in `ShopLook`'s "Removed" section with a Restore button.
- **Image sizing uses a JS-measured pixel box, not CSS `max-h-full` shrink-wrap — on every breakpoint, not just desktop.** The canvas stage now has a FIXED height everywhere (`h-[65dvh]` on mobile since the page scrolls below it, `flex-1` filling the immersive column on desktop), so a portrait photo needs to fit both axes while hotspots stay pixel-accurate. `max-height: 100%` on an image inside an auto-sized (shrink-wrapped) wrapper is genuinely ambiguous per the CSS spec — percentage heights don't resolve against a container whose own height comes from its content. `RestyleCanvas.tsx` (and `r/[id]/ShareCanvas.tsx`, which mirrors it, desktop-only for now) instead: watches the stage via `ResizeObserver`, computes the contained image's exact pixel box from the actual rendered `<img>`'s `naturalWidth`/`naturalHeight` (measured on load — preferred over trusting `restyle.width`/`height` alone, which is only the instant-first-paint fallback before the image loads), and pins the wrapper to that box via inline `style` (`position:absolute; left/top/width/height` in px). Hotspots/popovers keep their existing %-position code unchanged — they just land correctly because the wrapper IS the image's true rendered box, in pixels, not a CSS approximation. Falls back to plain `w-full h-auto` before the first measurement lands. Behind this sits a blurred, darkened `object-cover` copy of the same image (`blur-2xl brightness-50 scale-110`) filling the STAGE edge-to-edge (no rounding, no shadow, no border — it's just backdrop) — also now universal, not desktop-only. The sharp photo itself floats on top as a rounded, shadowed card (`rounded-3xl shadow-[var(--shadow-pop)] overflow-hidden` on `imgWrapClass`/`FALLBACK_WRAP`) — this is the RoomSketch-style look the user asked for after initially trying (and rejecting) a fully edge-to-edge un-rounded photo. On mobile, `RestyleStudio.tsx` renders the whole stage full-bleed ABOVE the padded column that holds QueuedChanges/ShopLook/VersionsStrip, not inside that padding.
- **The "+ Add" flow asks for location, then what it is, then how to source it — in that order.** `useRestyleWorkspace`'s `startAddFlow()` (triggered by the floating "+ Add" button) immediately shows `PinPlacementLayer` over the original photo via `pinRequest = {editId: null, label: ""}` — `editId: null` is the signal (to `RestyleCanvas` and to `PinPlacementLayer`'s handlers) that no edit exists yet, so placing/skipping a pin here (`placeAddLocation`/`skipAddLocation`) just stores `{x,y,note}` in `pendingAddPlacement` rather than PATCHing a real edit. Sourcing then opens with an empty label, and `SourcePanel` gates its normal tabs behind a one-line "What are you adding?" text step (`ws.setSourcingLabel`) whenever `mode==="add" && !label` — a fresh add-flow item always arrives here with an empty label, so this step always shows. Once the item is actually sourced (link/photo/description — all three tabs are available for "add" now, since a real label exists before you ever reach them) and the edit is created, `finalizeAddPlacement(editId, label)` attaches `pendingAddPlacement` immediately if one was captured, or falls back to the OLD post-hoc `requestPin` prompt if location was skipped — so a user who skips still gets one more chance right after the item lands. `pinRequest.editId: string` (non-null) is still what that post-hoc/legacy path uses, and is also how `setPlacement` (a real PATCH) is invoked from `RestyleCanvas`'s pin-layer branch — the two cases are told apart there by checking `ws.pinRequest.editId`.
- **"Find similar" works for ANY detected item, not just ones already swapped/staged.** `SimilarItemsPanel` used to only search off an already-staged edit's `reference_url` — a never-touched item had nothing to search from. `runVisualSearchByUrl` now takes an optional `box2d` (a detected object's own `box_2d`); when passed alongside the ORIGINAL photo's URL, `visual-search/route.ts` crops the original down to just that item server-side (`cropToBox`, same helper screenshot cleanup uses) before running the normal Lens+keyword pipeline — one shared code path either way. Entry point: `SourcePanel.tsx` shows a "Find similar items" button for any swap target with a detected match (`sourcing.mode==="swap" && matchedObject`), not just after staging something.
- A camera-captured **portrait** photo in `/restyle/new` can't be forced landscape by a web file input (no access to lens/zoom) — the capture screen shows a tip instead, and a portrait result triggers a retake prompt with an explicit "Use it anyway" escape hatch. Library uploads (existing photos) are never nagged — only the "Take a photo" path checks dimensions.
- `ProgressOverlay` (ui.tsx) is **self-ticking**: given `startedAt` + `expectedSeconds`, it runs its own 300ms interval internally to compute %, "~Xs left", and the rotating warm message index. This used to live in `useRestyleWorkspace` and re-render the ENTIRE editor tree every 300ms during a generate — same class of bug as the documented compare-slider fix (CLAUDE.md's original callout for that). If a similar "tick something during a long operation" need comes up again, keep the ticker local to the one component that displays it, not the shared hook.
